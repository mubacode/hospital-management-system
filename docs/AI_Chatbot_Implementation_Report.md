# AI Chatbot Controller: Implementation & Architecture Document
**Project:** Online Hospital Management System - AI Customer Representative integration

## 1. Executive Summary
The goal of this implementation was to replace a rigid, button-based navigational chatbot with a highly intelligent, natural-language, task-oriented **AI Customer Representative** for the Patient Panel. 

The resulting architecture successfully utilizes a massive 120-Billion Parameter Open Source Large Language Model (LLM) proxying securely through OpenRouter. Crucially, the AI is not just a text-generator; it utilizes strict **Function Calling (Tools)** to autonomously orchestrate complex multi-step backend database interactions—such as fetching doctors, verifying available dates, and directly executing SQL insertions to book or reschedule appointments.

---

## 2. Architecture & Technology Stack

*   **Frontend (UI):** React JS (`ChatAssistant.js`). A custom messaging interface built to parse markdown generated by the AI agent and instantly relay user intentions securely to the backend.
*   **Backend Interface:** Express JS / Node.js (`chatbotController.js`).
*   **Database:** MySQL (Managing `appointments`, `doctors`, `clinics`, and `patients`).
*   **LLM Provider API:** OpenRouter API (`openai/gpt-oss-120b:free`).
*   **API Wrapper:** Standard official `openai` NodeJS SDK.

## 3. Core Implementation Mechanics

### 3.1. Shifting to the OpenRouter/OpenAI Standard
Rather than relying on proprietary, vendor-locked SDKs, the system leverages the globally recognized OpenAI JSON-Schema standard. By pointing the baseline URL to `https://openrouter.ai/api/v1` instead of OpenAI's endpoint, the hospital avoids vendor lock-in. The system interacts effortlessly with the `openai/gpt-oss-120b:free` model, providing enterprise-grade intelligence entirely for free.

### 3.2. Action-Based Capabilities (Function Calling / Tool Use)
The most critical capability of the Chatbot Controller is that it can manipulate the hospital system. 
The LLM is initialized with an array of strictly defined JSON schemas known as "Tools." When the patient makes a request, the LLM analyzes the text and, instead of replying with text, actively asks the backend Server to run a specific tool. 

The following functional tools were developed for the LLM to access:
1.  `get_clinics`: Reads departments from the DB.
2.  `get_doctors`: Filters doctors by `clinicId`.
3.  `check_availability`: Calculates available times by scanning `appointments` for conflicts against a 9 AM - 5 PM slot generator.
4.  `book_appointment`: Inserts a `pending` appointment recursively checking for booking conflicts.
5.  `get_my_appointments`: Reads active patient appointments.
6.  `reschedule_appointment`: Updates the `appointment_date` and `appointment_time` dynamically.
7.  `cancel_appointment`: Updates the appointment `status` to 'cancelled'.

### 3.3. Advanced Conversational Memory & `reasoning_details` Loop
LLM REST APIs are fundamentally stateless. To hold a coherent conversation where a patient says *"I want to see a cardiologist tomorrow"*, the Controller orchestrates a persistent memory session per patient (`activeChats[userId]`).

Furthermore, because complex actions (like booking) require multiple sequential backend calls (e.g., getting clinics -> getting doctors -> checking times), the controller intercepts multi-step tool calls natively. 
Notably, the 120b Open Source model emits its internal logical tracing via the `reasoning_details` array. The Node controller is engineered to carefully extract this "thought process" and explicitly append it back into the persistent history array. This guarantees that the AI never loses its context when bouncing between the user, the AI engine, and the MySQL database.

## 4. Conversation Flows & Intent Mapping

The AI was instructed aggressively via a **System Instruction Protocol** which mandates strict constraints to emulate an actual Hospital Receptionist:
*   **Booking Requirement Enforcement:** The AI will refuse to book an appointment unless it has actively queried the `check_availability` database tool to confirm the requested window is unbooked.
*   **Rescheduling Protections:** The AI is instructed to force the user to first query their existing reservations (`get_my_appointments`) before arbitrarily changing times.
*   **Domain Isolation:** The System instruction explicitly orders the agent: *"If a user tries to converse about topics entirely unrelated to healthcare or the hospital, politely redirect them back to the hospital services."* This prevents edge-case prompt injection or general Q&A usage.

## 5. Security & Error Handling

*   **Blind Database Interaction:** The LLM does *not* write SQL. It merely outputs a JSON string requesting an action (e.g., `{ doctorId: 4, date: "2026-05-12" }`). The Node backend parses this safely, sanitizing inputs and handling the direct SQL `executeTool()` layer independently to protect against SQL injections via chat.
*   **HTTP 429 Graceful Handlers:** If the upstream upstream AI network provider throttles service due to global traffic overload, the server actively traps the `429` Rate Limit HTTP code to return a user-friendly message (`"I'm currently hitting OpenRouter rate-limits. Please wait a minute and try again."`) instead of crashing the Node instance.
